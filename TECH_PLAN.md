## Фаза 0 — Підготовка (1–2 тижні)

Спільне для обох сторін.

- Налаштування Azure: App Service, PostgreSQL Flexible Server, Blob Storage, Key Vault
- Репозиторій та CI/CD pipeline (GitHub Actions → Azure)
- Реєстрація застосунку в LinkedIn Developer Portal та Facebook Developers для OAuth
- Реєстрація акаунту GetStream, створення застосунку, налаштування push провайдерів (APNs + FCM) у GetStream Dashboard
- Визначення структури репозиторію (monorepo або окремі репо для backend / mobile / admin)

## Фаза 1 — Авторизація та профіль (2–3 тижні)

### Backend

- OAuth 2.0 flow для LinkedIn і Facebook (отримання токену, підтягування базового профілю)
- Refresh token логіка (видача, оновлення, інвалідація)
- Endpoint'и профілю: `GET /users/me`, `PUT /users/me` (ім'я, біо, інтереси, місто). На клієнті **онбординг S2–S4 не викликає** `PUT`; після OAuth — `GET`, потім умовний `PUT` з локальної чернетки, якщо профіль на сервері ще неповний.
- Завантаження аватару в Azure Blob Storage (`POST /users/me/avatar`)
- Синхронізація користувача з GetStream при реєстрації (створення GetStream user з нашим user ID)
- Міграція БД: таблиці `users`, `interests`, `user_interests`, `cities`

### Frontend (Flutter)

- **`API_MODE`**: `mock` | `live` | `auto` (через `--dart-define`). У режимі **mock** REST-репозиторії та шлях токена Stream не б’ють мережу; **live** завжди використовує HTTP; **auto** — mock, якщо `API_BASE_URL` містить `example.com`, інакше live. Деталі в `specs/001-event-networking-app/quickstart.md`.
- Проєкт з підтримкою l10n з першого дня (українська + англійська)
- Базова навігація: Tab bar (4 вкладки), роутер
- Екрани онбордингу S1–S4: **S1 Welcome** (опис продукту, кнопки «Почати» → S2 та «Увійти» → окремий OAuth-екран), **S2–S4** — Ім'я, Інтереси, Місто
- OAuth flow через `flutter_appauth`
- Зберігання токенів: `flutter_secure_storage`
- Екран профілю (Tab 4): фото, ім'я, посилання, інтереси, рейтинг

## Фаза 2 — Події (2–3 тижні)

### Backend

- CRUD для подій з урахуванням ролі: адмін створює одразу, учасник — зі статусом `pending`
- `POST /events/{id}/attend` — підписка на подію
- `DELETE /events/{id}/attend` — скасування підписки (каскадне закриття доступу до чатів)
- Фільтрація списку подій: за інтересами + географічна близькість (Haversine на рівні SQL)
- Пагінація (`cursor-based` або `offset`)
- Пошук за назвою, тегом, містом (`GET /events?q=...&city=...&interest=...`)
- Міграція БД: таблиці `events`, `event_attendees`, `event_interests`

### Frontend (Flutter)

- Головний екран (Tab 1): список карток подій з пагінацією (infinite scroll)
- Стан гість / авторизований (різне відображення учасників)
- Картка події: всі поля, кнопка "Податись", список учасників
- Карта подій (Tab 2): Google Maps або Mapbox, відображення подій + локальних чатів міст
- Фільтри на карті (за датою, інтересами)
- Пошук подій

## Фаза 3 — Учасники та чати (3–4 тижні)

### Backend

- `GET /events/{id}/attendees` — список учасників (тільки для підписників)
- `POST /users/{id}/block` — блокування користувача
- `POST /events/{id}/trips` — створення локального чату поїздки (проксує запит у GetStream)
- `POST /trips/{id}/join` — запит на вступ у локальний чат
- `POST /trips/{id}/approve/{userId}` — схвалення запиту (з логікою: автоматично якщо те саме місто, вручну — якщо інше)
- Webhook від GetStream для синхронізації подій чату
- Міграція БД: таблиці `trips`, `trip_members`, `trip_join_requests`, `blocked_users`

### Frontend (Flutter)

- Tab 3 — список усіх чатів через GetStream Flutter SDK (загальні, локальні, DM)
- Розділ "Поїздки" на сторінці події: список локальних чатів з маршрутом і часом виїзду
- Екран створення локального чату (маршрут, час, роль: Водій / Пасажир / Потяг / Свій варіант)
- Відображення запитів на вступ та їх статусу
- Попередження про низький рейтинг поряд із запитом (якщо рейтинг < порогу)
- DM між учасниками (GetStream out of the box)

## Фаза 4 — Push-сповіщення та рейтинг (1–2 тижні)

### Backend

- Scheduler (Hangfire або Azure Functions) для постподійного фідбеку: через N годин після завершення події — push через GetStream API
- Кастомні push-тригери (поза GetStream чатом):
  - Хтось з твого міста підписався на ту ж подію
  - Нова подія за інтересами користувача
- `POST /events/{id}/attendees/{userId}/review` — залишити відгук (зірки 1–5 + текст)
- `GET /users/{id}/reviews` — список відгуків користувача
- Міграція БД: таблиці `reviews`, `push_preferences`

### Frontend (Flutter)

- Обробка push-сповіщень у застосунку (foreground + background)
- Налаштування нотифікацій у профілі
- Екран фідбеку після події (зірки + текстовий відгук)
- Відображення рейтингу та відгуків у профілі користувача

## Фаза 5 — Адмін-панель (1–2 тижні)

### Backend

- Роль `admin` у системі, захист endpoint'ів через атрибут авторизації
- `GET /admin/events/pending` — список подій на схвалення
- `POST /admin/events/{id}/approve` / `/reject`
- `GET /admin/users` — список користувачів з фільтрами
- `POST /admin/users/{id}/ban`
- `GET /admin/reports` — список скарг
- `POST /admin/reports/{id}/resolve`

### Frontend (Веб, .NET)

- Razor Pages або Blazor (залежно від команди)
- Сторінки: список подій (pending / active / archived), картка події з можливістю редагування
- Список заявок на додавання події від учасників
- Список користувачів (перегляд профілю, блокування)
- Список скарг з можливістю відмітити як вирішені

## Ключові ризики

**GetStream ліміти.** Безкоштовний план — 25 MAU. Для пілоту на одній IT-конференції вистачить. Перед масштабуванням — перейти на платний план.

**OAuth LinkedIn.** LinkedIn вимагає окремого запиту на верифікацію застосунку для доступу до email та профілю. Подавати заявку на Фазі 0, бо розгляд займає час.

**Haversine vs PostGIS.** Для MVP достатньо простої формули відстані між координатами міст на рівні SQL. PostGIS — при необхідності складніших геозапитів у майбутньому.

**GetStream push + кастомні тригери.** GetStream автоматично надсилає push для повідомлень у чаті. Кастомні тригери (нова подія, підписка з міста) — окремо через FCM або GetStream Custom Events API.

- Розважальні події
- Статус "Цікавить" (спостерігач)
